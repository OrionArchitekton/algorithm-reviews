## One-liner

Paste a claim, a "best X of 2026" listicle, a product page, or a vendor pitch.
A **governed agent** fans out across the **live web**, decides which sources are
**admissible** (fail-closed), runs one verify→rebut pass, and ships a **signed
review receipt**: verdict, per-claim confidence, timestamped live citations,
dissent notes, and a tamper-evident hash — while a glass-box panel shows exactly
which sources it **admitted (green)** vs **rejected (red)** and *why*.

## What makes this cool (the "whoa")

It is "an algorithm that reviews" whose **own reviewing is itself reviewable**.
The self-referential read is the deepest possible interpretation of the
`algorithm.reviews` domain. The glass-box admit/reject feed is the moment a
senior engineer leans in: this isn't a chatbot summarizing the web, it's a
governance layer adjudicating evidence in public, in real time, with a receipt
you can defend.

## Problem Statement

AI answers and online reviews are **unauditable**. You cannot tell a grounded
verdict from a hallucinated one; sources are stale, fabricated, or cherry-picked;
and "AI fact-checkers" are themselves opaque black boxes. Buyers, journalists,
compliance/procurement, and due-diligence teams need a verdict they can **defend**
with a citation trail and a timestamp — not a vibe. The status quo is 10 browser
tabs + manual judgment + no record of how the conclusion was reached.

## Why this builder

This is a near-perfect projection of the builder's day job — a two-plane
architecture (governance plane that decides admissibility + execution plane that
acts), fail-closed enforcement, tamper-evident receipts, evals, MCP — onto a
consumer-legible product. To a judge panel of senior engineers (Google,
Snowflake, PayPal, Capital One, Microsoft, LinkedIn) it reads as **credible, not
costumed**.

## Premises (Phase 3 — confirmed)

1. A senior-eng judge panel rewards **visible governance/audit** over raw answer
   prettiness. The product's value is the admit/reject reasoning, not the prose.
2. The live-web dependency must be **load-bearing and visible**, or it's just a
   chatbot. Admit/reject of live, timestamped sources is the proof.
3. **One verify + one rebut pass** is enough signal. Multi-round debate is
   scope-death — hard-capped.
4. **The fallback web provider must be a real deployable web API** (Jina Reader /
   Brave / Tavily) — NOT Claude Code's WebSearch/WebFetch, which do not exist in a
   deployed Vercel runtime. (Correction to the ideation note.)
5. The **LLM API key is the one true blocker** for a live demo. Mitigated by
   designing the model layer to accept either a raw Anthropic key **or** the
   Vercel AI Gateway.

## Architecture — two planes, one Next.js app

```text
Input (claim / listicle / URL / vendor pitch)
        │
        ▼
[1] Claim decomposition            ── execution plane (LLM: fast model)
        │   atomic, checkable claims
        ▼
[2] Fan-out research               ── execution plane
        │   per claim: generate queries → WebProvider.search → candidate sources
        ▼
[3] ADMISSIBILITY  (fail-closed)   ── GOVERNANCE PLANE  ◀── the differentiator
        │   ADMIT / REJECT each source with a reason:
        │   reject if stale> threshold · low-authority/unsourced · irrelevant ·
        │   unverifiable/paywalled · self-referential/circular
        ▼
[4] Extract admitted sources       ── execution plane
        │   WebProvider.extract → evidence snippet + fetch timestamp
        ▼
[5] Verify → Rebut (1 pass each)   ── GOVERNANCE PLANE
        │   verify: verdict from admitted evidence
        │   rebut:  actively try to refute with dissenting/admitted evidence
        ▼
[6] Review Receipt                 ── governance plane
            verdict · per-claim confidence · admitted citations (url+ts+quote) ·
            rejected sources+reasons · dissent note · SHA-256 receipt hash
```

Fail-closed rule: a claim with **no admissible evidence** resolves to
**"Unverifiable"**, never "False". Absence of proof ≠ proof of absence.

## Swappable WebProvider interface (the de-risk)

```ts
interface WebProvider {
  name: string
  search(query: string, opts?): Promise<SourceCandidate[]>   // {url,title,snippet,publishedAt?}
  extract(url: string): Promise<ExtractedDoc>                 // {url,markdown,fetchedAt,status}
}
```

- **Primary: `NimbleProvider`** — REST `POST https://sdk.nimbleway.com/v1/{search,extract}`,
  Bearer token. (The sponsor prize provider.)
- **Fallback: `JinaProvider`** — keyless: `r.jina.ai/<url>` for extract,
  `s.jina.ai/?q=` for search. Lets us build + demo before the Nimble key lands,
  and is a genuine graceful-degradation story for the judges.
- Provider chosen by env (`WEB_PROVIDER=nimble|jina`), with automatic fallback on
  provider error so the live demo never hard-fails.

## Model layer

- Vercel AI SDK (`ai` + `@ai-sdk/anthropic`) for streaming + structured outputs.
- Adjudication (admissibility, verify/rebut): **Claude Opus**. Decomposition +
  query generation: **Claude Haiku/Sonnet** (cheap/fast).
- Accepts `ANTHROPIC_API_KEY` **or** Vercel AI Gateway credentials — whichever the
  operator provides. No raw key in the repo; env only.

## Streaming "glass box" UX

- Server route emits a typed event stream (SSE / AI SDK data stream):
  `claim_parsed · query_fired · source_found · admit · reject · extracted ·
  verdict · receipt`.
- Left/main: the **live admit/reject feed** (green ADMIT / red REJECT, each with a
  one-line governance reason and the source).
- Right/bottom: the **Review Receipt card** — verdict chip, confidence bars,
  clickable timestamped citations, dissent note, copyable hash.
- **Re-verify** re-runs the live sweep; when fresh evidence moves the verdict the
  **hash changes** — the on-stage proof that it reads the live web, not memory.

## Addendum — locked technical decisions (post adversarial review, 2026-06-06)

Adversarial review scored the original plan 5/10 (→7.5/10 with these fixes); two
research passes resolved the unknowns. These decisions SUPERSEDE any conflicting
note above.

1. **Hard caps in code (not prompt)** — per run: ≤3 claims, ≤2 queries/claim,
   ≤8 total extracts, provider concurrency 2, cache extracts by URL. This is the
   single most important change: it keeps the run inside rate limits AND the
   Vercel function budget, and protects the live demo. Re-verify reuses the cache.
2. **LLM via Vercel AI Gateway (OIDC)** — primary path. Use a plain string model
   id (`anthropic/claude-opus-4.8` for adjudication, `anthropic/claude-haiku-4.5`
   for decomposition/bulk classification). In prod on Vercel, `VERCEL_OIDC_TOKEN`
   auto-injects → **no `ANTHROPIC_API_KEY` required**. Local dev uses
   `AI_GATEWAY_API_KEY` or `vercel env pull`. Optional override: if
   `ANTHROPIC_API_KEY` is set, use the `@ai-sdk/anthropic` provider directly.
3. **AI SDK 6 patterns** — `createUIMessageStream` + typed `data-*` parts for the
   admit/reject/verdict/receipt event stream; client `useChat({ transport:
   new DefaultChatTransport({ api }) })`; structured outputs via
   `Output.object({ schema })` (NOT deprecated `generateObject`).
4. **Real signature, not a bare hash** — receipts are signed with **ECDSA P-256**
   (Web Crypto `crypto.subtle.sign`). Private key from env (`RECEIPT_SIGNING_JWK`,
   generated once); public key published at `/api/pubkey`; a `/verify` view (and
   function) recomputes the canonical payload and verifies the signature. The
   canonical signed payload EXCLUDES volatile fields (fetch timestamps, source
   ordering) so it is deterministic — it changes only when evidence/verdict
   changes. Language: "signed review receipt" is now literally true; we also keep
   a content digest for display.
5. **Web providers (corrected)** — Primary `NimbleProvider`:
   `POST https://sdk.nimbleway.com/v1/{search,extract}`, `Authorization: Bearer`,
   confirmed current (ignore legacy `*.webit.live`). Validate with one curl before
   trusting it. Fallback for BUILD-before-key: `r.jina.ai/<url>` is keyless for
   extract; **`s.jina.ai` now requires a key (401 keyless)** so fallback search =
   keyless DuckDuckGo Lite HTML read through `r.jina.ai` (no key), with an optional
   `JINA_API_KEY`/Nimble upgrade. Provider auto-falls-back on error so the demo
   never hard-fails.
6. **Streaming on Vercel** — set `export const maxDuration = 300` and
   no-buffering headers (`X-Accel-Buffering: no`, `Cache-Control:
   no-store, no-transform`) on the stream route; the AI SDK data-stream handles
   keep-alive. **Verify streaming against the deployed URL at end of Day 1**, not
   on localhost only.
7. **Build sequence reorder** — Day-1 FIRST task: deploy a streaming hello-world
   route to Vercel and confirm incremental events render on the real `.vercel.app`
   URL. Build the agent loop into that proven shell.
8. **Adjudication is ONE call** — combined verify+dissent in a single Opus call
   returning verdict + confidence + strongest counter-evidence (keep the dissent
   output, cut the second round-trip). No multi-round debate.
9. **Guaranteed stage fallback** — a static receipt fixture committed to the repo,
   rendered by `/demo/[id]`. Zero external dependency; cannot rate-limit or 500.
